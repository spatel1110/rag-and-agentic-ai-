2.1 RAG Kya Hai — Poori Kahani

🧠 Ek senior engineer ki taraf se —

"Dekh bhai, main 5 saal se production RAG systems bana raha hoon. Sabse badi galti jo log karte hain woh yeh hai ki woh RAG ko ek magic black box samajhte hain. RAG ek engineering pattern hai — agar tune iska har component sahi se samjha, toh tu production-grade system bana sakta hai. Agar nahi samjha, toh tu chatbot banaayega jo kabhi kabhi sahi answer deta hai aur kabhi hallucinate karta hai."

Problem Before RAG — Kyun Zaroori Hua?

Dekh, jab LLMs (Large Language Models) jaise GPT-4, Claude, Gemini aaye, toh lagta tha ki yeh sab kuch jaante hain. Lekin reality mein unki ek fundamental problem hai — knowledge cutoff. ChatGPT ko train kiya gaya hai 2023 tak ke data pe. Toh agar tu pooche "meri company ka latest quarterly report kya kehta hai?" — toh woh hallucinate karega. Matlab confident hokar galat answer dega.

Doosri problem — private data. Teri company ke internal documents, policies, codebase — yeh sab LLM ke training data mein nahi hai. Toh LLM directly tera help nahi kar sakta in specific questions pe.

Teesri problem — hallucination. LLM bahut confidently galat information de deta hai. Medical, legal, financial domains mein yeh dangerous hai. Tu patient ko galat medicine suggest nahi kar sakta na?

RAG Ka Idea — Simple Lekin Powerful

RAG = Retrieval Augmented Generation. Isko samjhne ka sabse easy tarika hai exam analogy.

"Tu exam mein open book le jaata hai — woh open book hai tera Vector DB. Exam paper hai user ki query. Tu book mein relevant pages dhundhta hai (retrieval), phir un pages ki madad se answer likhta hai (generation). Tu hallucinate nahi karega kyunki answer tere saamne hai book mein!"

RAG Data Flow — Step by Step

Ab samjho ki RAG internally kaise kaam karta hai, har step detail mein:

Step 1: User query aati hai — "What is our company's leave policy?"
Step 2: Query ko embedding model se vector mein convert karte hain (1536 dimensional numbers)
Step 3: Vector DB mein similarity search — kaun se stored chunks is query ke sabse paas hain?
Step 4: Top-k relevant chunks retrieve hote hain (usually k=3 to 5)
Step 5: Retrieved chunks + original query = final prompt banate hain LLM ke liye
Step 6: LLM context mein answer generate karta hai — grounded, factual, no hallucination

Kyun Kaam Karta Hai?

RAG kaam karta hai kyunki LLM ko hallucinate karne ki zaroorat hi nahi padti. Answer already context mein hai. LLM ka kaam sirf yeh reh jaata hai ki relevant information ko user-friendly format mein present kare. Yeh Swiggy delivery boy jaisa hai — usko khana banana nahi aata, lekin woh sahi restaurant se sahi khana teri doorstep pe le aata hai. Vector DB = restaurant, LLM = delivery mechanism, User = hungry customer.

Complete RAG Pipeline — Working Python Code

# RAG Pipeline - End to End Implementation
# Har line Hinglish mein commented hai tere liye

# Step 1: Required imports
from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import Chroma
from langchain.chains import RetrievalQA
from langchain.prompts import PromptTemplate

# Step 2: Document Loading
# Apna PDF load karo — yeh tera knowledge base hai
loader = PyPDFLoader("company_handbook.pdf")
documents = loader.load()  # Saare pages load ho jaayenge

# Step 3: Chunking — Document ko chhote pieces mein todo
# Yeh sabse CRITICAL step hai — galat chunking = galat answers
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,       # Har chunk mein max 1000 characters
    chunk_overlap=200,     # 200 characters overlap — context preserve karne ke liye
    length_function=len,   # Character count use karo
    separators=["\n\n", "\n", " ", ""]  # Pehle paragraphs pe split, phir lines pe
)
chunks = text_splitter.split_documents(documents)
print(f"Total chunks created: {len(chunks)}")  # Kitne chunks bane check karo

# Step 4: Embeddings — Text ko numbers mein convert karo
# OpenAI ka embedding model use kar rahe hain
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",  # Cost effective + good quality
    openai_api_key="your-api-key"
)

# Step 5: Vector Store — Embeddings ko store karo searchable format mein
# Chroma local vector DB hai — dev ke liye perfect
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db",  # Disk pe save hoga — restart pe bhi rahega
    collection_name="company_docs"
)

# Step 6: Retriever Setup — Search engine banao
retriever = vectorstore.as_retriever(
    search_type="similarity",  # Cosine similarity se search
    search_kwargs={"k": 4}     # Top 4 relevant chunks laao
)

# Step 7: LLM Setup — Answer generator
llm = ChatOpenAI(
    model="gpt-4o-mini",       # Fast + cheap + good quality
    temperature=0,              # Deterministic answers — no creativity needed
    openai_api_key="your-api-key"
)

# Step 8: Custom Prompt — LLM ko instructions do
prompt_template = """Tu ek helpful assistant hai. Neeche diye gaye context se answer de.
Agar context mein answer nahi hai, toh bol "Mujhe is baare mein information nahi mili."
Kabhi bhi apne se answer mat banaa (hallucinate mat kar).

Context: {context}

Question: {question}

Answer:"""

PROMPT = PromptTemplate(
    template=prompt_template,
    input_variables=["context", "question"]
)

# Step 9: Chain banao — Sab connect karo
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",  # Saare chunks ek saath LLM ko do
    retriever=retriever,
    chain_type_kwargs={"prompt": PROMPT},
    return_source_documents=True  # Source bhi return karo — transparency ke liye
)

# Step 10: Query karo!
result = qa_chain.invoke({"query": "What is our company's leave policy?"})
print("Answer:", result["result"])
print("Sources:", [doc.metadata["source"] for doc in result["source_documents"]])

⚠️ Common Mistakes Jo Engineers Karte Hain

1.

Chunk size galat set karna

Bahut bada chunk (5000+ chars) = irrelevant noise aayega. Bahut chhota chunk (100 chars) = context missing hoga. Sweet spot usually 500-1000 chars hai general RAG ke liye.

2.

Query aur Document embedding ke liye alag model use karna

Agar documents text-embedding-3-small se embed kiye hain toh query bhi SAME model se embed karna padega. Different models ka vector space alag hota hai — match nahi karega!

3.

k value bahut kam rakhna

k=1 rakh diya toh ek hi chunk milega — agar woh chunk partially relevant hai toh answer incomplete hoga. k=3 to 5 usually best hai. Bahut zyada (k=20) pe noise badh jaata hai.

2.3 Chunking Strategy — Yeh Galat Ho Toh Sab Kuch Galat

🧠 Ek senior engineer ki taraf se —

"Main tujhe sach bata raha hoon — meri pehli RAG system mein 80% problems chunking ki wajah se thi. Embedding model achha tha, LLM achha tha, lekin chunks galat the toh answers bhi galat aa rahe the. Chunking ko tune sahi kar liya toh aadhi battle jeet gaya."

Why Chunking Matters — Analogy Se Samjho

Soch tu ek kitaab ka summary banana chahta hai. Agar tu poori 500-page kitaab ek baar mein LLM ko de dega toh — pehle toh token limit mein fit nahi hogi, aur agar fit ho bhi gayi toh LLM confuse hoga itne text mein se relevant part dhundhne mein. Ab agar tu sirf ek line de — "Chapter 3 talks about..." — toh context hi nahi hoga, kuch samajh nahi aayega.

Chunking = sahi size ke pages banana. Na bahut bade, na bahut chhote. Aur overlap isliye rakhte hain ki agar koi important sentence do chunks ke beech mein aata hai, toh woh dono mein preserved rahe.

Zomato menu jaisa soch — agar tu poora menu ek image mein dikhaye (too big), toh customer confused. Agar sirf dish ka naam dikhaye bina description (too small), toh customer decide nahi kar paayega. Best = dish name + short description + price (right chunk size).

RecursiveCharacterTextSplitter — Default Choice

Yeh sabse smart splitter hai LangChain mein. "Recursive" ka matlab hai ki yeh hierarchy mein split karta hai. Pehle try karega paragraphs pe split karna (\n\n), agar chunk size se bada ho toh single newline pe (\n), phir spaces pe, aur last mein character by character. Iska matlab — yeh natural boundaries respect karta hai. Sentence beech mein nahi katega jab tak absolutely zaroori na ho.

chunk_size=1000: Har chunk max 1000 characters ka hoga

chunk_overlap=200: Consecutive chunks mein 200 chars common honge — yeh ensure karta hai ki boundary pe context lost na ho

CharacterTextSplitter — Simple But Risky

Yeh simplest hai — fixed size pe split kar deta hai. Problem: sentence beech mein cut ho sakta hai. "The company was founded in" | "1999 by Mr. Sharma" — ab pehla chunk incomplete hai. Use kab karo: jab tera document already well-structured ho (like log files, structured data).

TokenTextSplitter — Token-Aware Splitting

Characters nahi, tokens count karta hai. Kyun better? Kyunki LLM token limits mein kaam karta hai, character limits mein nahi. "Hello" = 1 token, but "Pneumonoultramicroscopicsilicovolcanoconiosis" bhi mostly 3-4 tokens. Token budget tight ho toh yeh use karo — direct mapping hoti hai LLM ki capacity se.

SemanticChunker — Advanced Meaning-Based

Yeh sabse intelligent hai — meaning ke basis pe split karta hai. Har sentence ko embed karta hai, phir dekhta hai ki consecutive sentences ka meaning kitna similar hai. Jahan pe meaning abruptly change hoti hai — wahan split karta hai. Three breakpoint types:

  • Percentile: Top X% dissimilar points pe split — predictable number of chunks
  • Standard Deviation: Mean se Z standard deviations door hone pe split — statistical approach
  • Gradient: Sudden change in similarity — most natural breaks detect karta hai

Kab use karo: Research papers, legal documents, jahan semantic coherence critical hai. Cost: slow hai kyunki har sentence embed karna padta hai.

# Chunking Strategies — Complete Comparison
from langchain.text_splitter import (
    RecursiveCharacterTextSplitter,
    CharacterTextSplitter,
    TokenTextSplitter
)
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

sample_text = """RAG systems have revolutionized how we build AI applications.
The key insight is that LLMs work much better when given relevant context.
This context comes from a retrieval step that searches through your documents.

The chunking strategy determines how documents are split into smaller pieces.
Each piece becomes a searchable unit in your vector database.
Choosing the right chunk size is critical for retrieval quality."""

# === RecursiveCharacterTextSplitter — Best Default Choice ===
recursive_splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,          # Max 200 chars per chunk
    chunk_overlap=50,        # 50 chars overlap between chunks
    separators=["\n\n", "\n", ". ", " ", ""],  # Split hierarchy
    length_function=len
)
recursive_chunks = recursive_splitter.split_text(sample_text)
print(f"Recursive: {len(recursive_chunks)} chunks")

# === CharacterTextSplitter — Simple Fixed Size ===
char_splitter = CharacterTextSplitter(
    chunk_size=200,
    chunk_overlap=50,
    separator="\n"  # Sirf newline pe split karega
)
char_chunks = char_splitter.split_text(sample_text)
print(f"Character: {len(char_chunks)} chunks")

# === TokenTextSplitter — Token-Aware ===
token_splitter = TokenTextSplitter(
    chunk_size=50,           # 50 TOKENS (not characters!)
    chunk_overlap=10         # 10 token overlap
)
token_chunks = token_splitter.split_text(sample_text)
print(f"Token: {len(token_chunks)} chunks")

# === SemanticChunker — Meaning-Based (Advanced) ===
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
semantic_splitter = SemanticChunker(
    embeddings=embeddings,
    breakpoint_threshold_type="percentile",  # or "standard_deviation", "gradient"
    breakpoint_threshold_amount=70  # Top 70% dissimilarity = break
)
semantic_chunks = semantic_splitter.split_text(sample_text)
print(f"Semantic: {len(semantic_chunks)} chunks")

2.4 Embeddings — Numbers Mein Meaning

🧠 Ek senior engineer ki taraf se —

"Embeddings samajhna RAG ki foundation hai. Jab tak tu yeh nahi samjhega ki text kaise numbers mein convert hota hai aur similar text ke numbers kyun paas hote hain, tab tak tu RAG debug nahi kar paayega. Main ne ek baar 2 hafte lagaye yeh debug karne mein ki answers galat kyun aa rahe — problem thi ki embedding model change kar diya tha bina re-indexing ke!"

Embedding Kya Hai — Deep Explanation

Embedding ka matlab hai text ko numbers ki ek list (vector) mein convert karna aise tarike se ki meaning preserve ho. "King" ka vector aur "Queen" ka vector paas honge kyunki dono royalty hain. "King" aur "Banana" ke vectors door honge kyunki unka meaning alag hai.

Famous example: King - Man + Woman ≈ Queen. Yeh isliye kaam karta hai kyunki embedding space mein "gender" ek direction hai. Agar tu "Man" ka vector minus kare aur "Woman" ka add kare, toh tu basically gender direction change kar raha hai — aur result "Queen" ke paas aata hai.

"Aise soch — Delhi ka pincode 110001 hai, Mumbai ka 400001. Dono cities hain but numbers alag hain. Embedding mein similar concepts ke 'pincodes' (numbers) close hote hain. Jaise South Delhi ke sab areas ke pincodes 1100XX range mein hain — waise hi similar text ke embeddings paas hote hain."

Distance Metrics — Similarity Kaise Measure Karein

Cosine Similarity

Vectors ka angle dekhta hai. Direction matter karti hai, length nahi. Agar do texts same topic pe hain toh cosine high hoga chahe ek chhota ho ya bada. Best for text — 90% cases mein yeh use karo.

Euclidean Distance

Actual distance measure karta hai 2 points ke beech. Magnitude sensitive — agar ek vector bada hai toh distance zyada dikhega even if direction same. Use for: normalized vectors, image embeddings.

Dot Product

Direction + magnitude dono consider karta hai. Fastest compute. But needs normalized vectors for fair comparison. Use for: when vectors are already normalized (most embedding APIs normalize).

# === Similarity Check — Manual ===
import numpy as np

def cosine_similarity(v1, v2):
    """Do vectors kitne similar hain — 1.0 = same, 0.0 = unrelated"""
    dot = np.dot(v1, v2)
    norm = np.linalg.norm(v1) * np.linalg.norm(v2)
    return dot / norm

2.5 Vector Stores — Production Mein Kaunsa Use Karo

🧠 Ek senior engineer ki taraf se —

"Dev ke liye Chroma use karo, prototype ke liye FAISS, production ke liye Qdrant ya Pinecone. Main personally Qdrant prefer karta hoon — self-host kar sakta hai, filtering powerful hai, aur performance solid hai. Pinecone tab use karo jab infra manage nahi karna ho."

Vector Store Internally Kaise Kaam Karta Hai

Vector store basically ek specialized database hai jo vectors (numbers ki lists) store karta hai aur unpe fast similarity search karta hai. Normal database mein tu exact match ya range query karta hai. Vector DB mein tu poochta hai "mujhe is vector ke sabse SIMILAR vectors do." Internally yeh Approximate Nearest Neighbor (ANN) algorithms use karta hai:

HNSW

Hierarchical Navigable Small World — graph-based. Fast search, high memory. Best for most production cases.

IVF

Inverted File Index — clusters mein divide karta hai. Less memory, slightly less accurate. Good for huge datasets.

Flat

Brute force — har vector se compare karo. 100% accurate but slow for large data. Good for small datasets (<100K).

3.1 Retrieval Techniques — Basic Se Advanced

🧠 Ek senior engineer ki taraf se —

"Simple similarity search se start karo, phir jab accuracy issues aayein toh MMR add karo, phir hybrid search, phir re-ranking. Har layer add karne ka matlab hai latency badhega aur cost badhega — toh blindly sab mat add karo. Measure karo pehle ki problem kya hai, phir solve karo."

MMR — Maximal Marginal Relevance

Problem: Bina MMR ke agar tu "Python best practices" search kare, toh top 5 results almost same content ke honge — sab "use virtual environments" ke baare mein. Kyunki woh sabse relevant hai. Lekin user ko diverse answers chahiye — virtual env, type hints, testing, etc.

MMR ka formula: Score = λ × Relevance(doc, query) - (1-λ) × max(Similarity(doc, already_selected_docs))

λ (lambda) = 1.0 means pure relevance (no diversity). λ = 0.0 means maximum diversity. Sweet spot usually 0.5-0.7.

3.2 Advanced RAG Patterns — Production Architecture

Pattern 1: RAG Fusion

RAG Fusion Multi-Query ka advanced version hai. Difference: Multi-Query sirf results combine karta hai (union), RAG Fusion Reciprocal Rank Fusion (RRF) use karta hai ranking ke liye. RRF ka idea simple hai — agar ek document multiple queries mein top pe aata hai, toh woh definitely important hai. Score = sum(1 / (rank + k)) across all queries.

Indian analogy: Jaise JEE mein Physics, Chemistry, Maths teen subjects hain. Agar ek student teeno mein top 10 mein hai, toh woh overall top mein hoga. RRF wahi karta hai — multiple rankings ko combine karta hai.

flowchart TD Q["Original Query"] --> LLM["LLM generates 4 variations"] LLM --> Q1["Query 1"] LLM --> Q2["Query 2"] LLM --> Q3["Query 3"] LLM --> Q4["Query 4"] Q1 --> VDB["Vector DB Search each"] Q2 --> VDB Q3 --> VDB Q4 --> VDB VDB --> RRF["Reciprocal Rank Fusion"] RRF --> TOP["Top Ranked Documents"] TOP --> GEN["LLM Generate Answer"] 
# RAG Fusion — Multiple Queries + RRF Ranking
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)

# Step 1: Generate query variations
query_gen_prompt = ChatPromptTemplate.from_template(
    """Tu ek helpful assistant hai. User ki query ke 4 alag perspectives generate kar.
    Har variation alag angle se same information dhundhne ki koshish kare.
    
    Original query: {question}
    
    4 alternative queries (ek per line):"""
)

def generate_queries(question):
    response = llm.invoke(query_gen_prompt.format(question=question))
    return [q.strip() for q in response.content.split("\n") if q.strip()]

# Step 2: Search each query
def search_all_queries(queries, retriever):
    all_results = {}
    for query in queries:
        docs = retriever.invoke(query)
        all_results[query] = docs
    return all_results

# Step 3: Reciprocal Rank Fusion
def reciprocal_rank_fusion(results_dict, k=60):
    """RRF formula: score = sum(1 / (rank + k)) for each query's ranking"""
    fused_scores = {}
    for query, docs in results_dict.items():
        for rank, doc in enumerate(docs):
            doc_id = doc.page_content[:100]  # Use content as ID
            if doc_id not in fused_scores:
                fused_scores[doc_id] = {"doc": doc, "score": 0}
            fused_scores[doc_id]["score"] += 1 / (rank + k)
    
    # Sort by fused score — highest first
    ranked = sorted(fused_scores.values(), key=lambda x: x["score"], reverse=True)
    return [item["doc"] for item in ranked[:5]]  # Top 5

# Full pipeline
question = "What are the key benefits of RAG systems?"
queries = generate_queries(question)
all_results = search_all_queries(queries, retriever)
final_docs = reciprocal_rank_fusion(all_results)

Pattern 2: HyDE (Hypothetical Document Embeddings)

Yeh genius trick hai. Problem: User ki query chhoti hoti hai ("What is RAG?") aur documents bade hote hain (500 words ka paragraph). Short query ka embedding aur long document ka embedding ka semantic gap hota hai — match ache se nahi hota.

Solution: Pehle LLM se ek hypothetical answer generate karo (without retrieval), phir US answer ko embed karo aur search karo. Hypothetical answer document-length ka hoga, toh vector space mein real documents ke paas hoga. Jaise tu interview mein jaane se pehle expected questions ke model answers prepare karta hai — wahi concept.

Kab use karo: Jab simple queries pe poor retrieval ho rahi ho. Kab NAHI: Factoid queries pe (jahan query already specific hai). Trade-off: Extra LLM call = more latency + cost.

Pattern 3: Corrective RAG (CRAG)

Normal RAG ki ek badi problem hai — agar retrieved documents relevant nahi hain toh bhi LLM unse answer generate kar dega (garbage in = garbage out). CRAG kehta hai: Retrieve karo, phir GRADE karo ki documents relevant hain ya nahi. Agar nahi hain, toh web search fallback pe jao.

Analogy: Tu exam mein open book use kar raha hai. Agar tere book mein answer nahi mil raha, toh tu chupke se Google karta hai (web search). CRAG yahi karti hai — systematically.

flowchart TD Q["User Query"] --> R["Retrieve from Vector DB"] R --> G["Grade Each Document (LLM)"] G --> D{"Relevance Check"} D -->|"All Relevant"| GEN["Generate Answer"] D -->|"Some Ambiguous"| WEB["Supplement with Web Search"] D -->|"All Irrelevant"| RW["Rewrite Query + Web Search"] WEB --> GEN RW --> GEN GEN --> OUT["Final Answer"] 
# Corrective RAG (CRAG) — Self-Correcting Retrieval
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# Document Grader — kya yeh document query ke liye relevant hai?
grade_prompt = ChatPromptTemplate.from_template(
    """Tu ek grader hai. Evaluate kar ki yeh document user ki query ke liye relevant hai ya nahi.
    Sirf 'yes' ya 'no' bol.
    
    Document: {document}
    Question: {question}
    
    Relevant (yes/no):"""
)

def grade_document(doc, question):
    """Ek document grade karo — relevant hai ya nahi"""
    response = llm.invoke(grade_prompt.format(
        document=doc.page_content, question=question
    ))
    return response.content.strip().lower() == "yes"

def corrective_rag(question, retriever):
    """Full CRAG pipeline"""
    # Step 1: Retrieve
    docs = retriever.invoke(question)
    
    # Step 2: Grade each document
    relevant_docs = []
    for doc in docs:
        if grade_document(doc, question):
            relevant_docs.append(doc)
    
    # Step 3: Decision
    if len(relevant_docs) >= 2:
        # Enough relevant docs — generate directly
        context = "\n".join([d.page_content for d in relevant_docs])
    elif len(relevant_docs) == 1:
        # Partial — supplement with web search
        web_results = web_search(question)
        context = relevant_docs[0].page_content + "\n" + web_results
    else:
        # No relevant docs — rewrite query + web search
        rewritten = rewrite_query(question)
        context = web_search(rewritten)
    
    # Step 4: Generate answer
    return generate_answer(context, question)

Pattern 4: Self-RAG (Self-Reflective RAG)

Self-RAG sabse advanced pattern hai. Agent khud decide karta hai har step pe: (1) Kya mujhe retrieve karna chahiye? (2) Jo retrieve kiya woh relevant hai? (3) Mera answer documents se faithful hai? (4) Mera answer useful hai? Agar kisi step pe "no" aata hai toh woh loop back karta hai. Yeh 4 "reflection tokens" hain — ISRET, ISREL, ISSUP, ISUSE.

Soch aise — tu report likh raha hai. Pehle sochta hai "kya mujhe reference chahiye?" (ISRET). Phir reference padhta hai aur sochta hai "yeh relevant hai?" (ISREL). Report likhta hai aur check karta hai "mera likha hua reference ke saath match karta hai?" (ISSUP). Finally "kya yeh helpful hai reader ke liye?" (ISUSE).

(ISREL)"} GR -->|"Yes"| GEN2["Generate with Context"] GR -->|"No"| TQ["Transform Query & Retry"] TQ --> RET GEN2 --> GH{"Is answer faithful? (ISSUP)"} GH -->|"Yes"| USE{"Is answer useful? (ISUSE)"} GH -->|"No"| GEN2 USE -->|"Yes"| OUT["Output Final Answer"] USE -->|"No"| TQ

Pattern 5: Graph RAG — Knowledge Graphs + RAG

Normal RAG mein ek chunk se answer milta hai. Lekin kuch questions ke liye multiple hops chahiye. "Elon Musk ki company ka CEO kaun hai?" — pehle Elon Musk → Tesla/SpaceX (entity), phir Tesla → CEO (relation). Single chunk mein yeh sab nahi hoga necessarily.

Graph RAG mein documents se entities aur relationships extract karte hain, Knowledge Graph (Neo4j) mein store karte hain, aur query time pe graph traversal se multi-hop answers milte hain. Expensive hai — lekin complex domains (healthcare, finance, legal) mein game-changing hai.

🧠 Ek senior engineer ki taraf se —

The Fundamental Difference

RAG: Query → Retrieve → Answer. Ek shot. Ek baar. Done. Koi decision-making nahi. Koi loop nahi. Tu poochh, woh jawab de, finish.

Agent: Query → Think → Plan → Act → Observe → Think again → Act again → ... → Final Answer. Multiple steps. Decisions har step pe. Self-correction possible.

"RAG ek fast food counter hai — tu order de, 2 minute mein burger mil gaya, chal nikal. Agent ek personal chef hai — pehle poochta hai tera taste kya hai, allergy hai kya, phir recipe plan karta hai, ingredients check karta hai fridge mein, agar nahi hai toh market jaata hai, cook karta hai, taste karta hai, adjust karta hai, phir serve karta hai. Quality difference samajh aaya?"

What Makes Something an Agent — 5 Components

🧠 1. LLM as Brain

Reasoning engine. Decisions leta hai. Plan banata hai. Natural language samajhta hai. GPT-4o, Claude — yeh brain hain agent ka.

🔧 2. Tools (Hands)

Actions jo agent le sakta hai. Web search, database query, API calls, code execution, file operations. Bina tools ke agent sirf soch sakta hai, kar kuch nahi sakta.

💾 3. Memory

What agent remembers. Short-term = current conversation. Long-term = past interactions. Bina memory ke agent goldfish hai — har turn pe bhool jaata hai.

📋 4. Planning

Complex task ko steps mein todna. "Research paper likhna hai" → outline → each section → review → finalize. Without planning agent randomly tools call karega.

👁️ 5. Observation Loop

Har action ke baad result observe karo, evaluate karo, next action decide karo. Yeh woh loop hai jo agent ko "intelligent" banata hai. Trial and error + learning.

5.1 LangGraph Kya Hai — Core Concepts

🧠 Ek senior engineer ki taraf se —

"LangGraph samajhne ke liye pehle samjho ki simple chains kyun limited hain. Chain mein A → B → C hota hai — ek direction, koi decision nahi, koi loop nahi. Real agents ko loops chahiye (retry), conditional logic chahiye (if/else), shared state chahiye (context sharing). LangGraph = graph jahan nodes functions hain aur edges decisions hain. Production agents ke liye yeh THE framework hai."

Core Concepts — Deep Dive

📦 STATE

Shared dictionary jisme agent ka poora context hota hai — messages, retrieved docs, decisions, intermediate results. TypedDict se define karo. Har node state ko read karta hai, process karta hai, aur updated state return karta hai. State = agent ki "working memory".

⚙️ NODES

Python functions jo graph ke building blocks hain. Har node ek kaam karta hai — retrieve, grade, generate, search. Input: current state. Output: state update. Nodes ko simple aur focused rakhna chahiye — ek node ek kaam.

↗️ EDGES

Nodes ke beech connections. Normal edge: hamesha A se B jao. Conditional edge: state dekh ke decide karo kahan jaana hai. Yeh woh jagah hai jahan decision-making hoti hai — "agar document relevant hai toh generate, nahi toh web search."

🚀 COMPILE & RUN

Graph define karne ke baad compile karo (validation hoti hai), phir invoke karo input state ke saath. Graph internally state ko nodes through flow karta hai jab tak END node na aa jaaye.

5.2 ReAct Pattern — Sabse Important

🧠 Ek senior engineer ki taraf se —

"ReAct (Reason + Act) sabse fundamental agent pattern hai. Agar tu sirf ek pattern seekhe poori life mein, toh yeh seekh. Har modern agent framework internally ReAct hi use karta hai. Idea simple hai: Think → Act → Observe → Repeat. Jab tak task complete na ho."

ReAct = Reasoning + Acting. 2022 mein Google ka paper tha jo kehta hai: Agar LLM ko sirf act karne bolo (tools call karo) without thinking — toh galat tools call karega. Agar sirf think karne bolo without acting — toh hallucinate karega. Dono combine karo — think THEN act THEN observe THEN think again. Yeh iterative loop agent ko powerful banata hai.

flowchart TD START(["👤 User Query"]) --> THINK["💭 Thought: What do I need to do?"] THINK --> ACT["⚡ Action: Call a tool"] ACT --> OBS["👁️ Observation: Tool result"] OBS --> THINK2{"🤔 Is task complete?"} THINK2 -->|"No, need more info"| THINK THINK2 -->|"Yes, have enough!"| ANS["✅ Final Answer"] 
# ReAct Agent — Full LangGraph Implementation
from typing import TypedDict, Annotated, Sequence
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI
from langchain_core.messages import BaseMessage, HumanMessage, ToolMessage
from langchain_community.tools.tavily_search import TavilySearchResults
from langchain.tools import tool
import operator

# === Tools Define Karo ===
search_tool = TavilySearchResults(max_results=3)

@tool
def calculator(expression: str) -> str:
    """Calculate a mathematical expression. Input: math expression like '2+2' or '100*0.15'"""
    try:
        return str(eval(expression))  # Production mein safe eval use karna!
    except:
        return "Error: Could not calculate"

tools = [search_tool, calculator]

# === State Definition ===
class ReActState(TypedDict):
    messages: Annotated[Sequence[BaseMessage], operator.add]

# === LLM with Tools ===
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
llm_with_tools = llm.bind_tools(tools)  # LLM ko tools ke baare mein batao

# === Node Functions ===
def call_model(state: ReActState) -> dict:
    """LLM call karo — woh decide karega tool use karna hai ya final answer dena hai"""
    messages = state["messages"]
    response = llm_with_tools.invoke(messages)
    return {"messages": [response]}

def should_continue(state: ReActState) -> str:
    """Check: LLM ne tool call kiya ya final answer diya?"""
    last_message = state["messages"][-1]
    # Agar tool_calls hain toh tools execute karo
    if last_message.tool_calls:
        return "tools"
    # Nahi toh final answer hai — END
    return "end"

# === Graph Build ===
workflow = StateGraph(ReActState)

# Nodes
workflow.add_node("agent", call_model)           # LLM thinks + decides
workflow.add_node("tools", ToolNode(tools))       # Tools execute

# Edges
workflow.set_entry_point("agent")
workflow.add_conditional_edges(
    "agent",
    should_continue,
    {"tools": "tools", "end": END}  # Tool call? → execute. No? → done.
)
workflow.add_edge("tools", "agent")  # Tool result wapas agent ko (LOOP!)

# Compile
react_agent = workflow.compile()

# === RUN IT! ===

})

5.3 Plan and Execute Pattern

ReAct mein agent har step pe decide karta hai. Plan-and-Execute mein pehle POORA plan banate hain, phir ek ek step execute karte hain. Agar koi step fail ho ya naya information mile, toh REPLAN karte hain. Complex multi-step tasks ke liye yeh better hai ReAct se — kyunki agent "big picture" pehle dekhta hai.

Jaise tu road trip plan karta hai — pehle route decide karo (Delhi → Agra → Jaipur → Udaipur), phir ek ek step execute karo. Agar Agra mein road block hai toh replan karo (via Mathura). Tu har km pe plan nahi change karta — high-level plan pehle bana, execute step by step.

flowchart TD Q["User Task"] --> PLAN["🗺️ Planner: Create step-by-step plan"] PLAN --> S1["Step 1: Execute"] S1 --> S2["Step 2: Execute"] S2 --> CHECK{"New info? Step failed?"} CHECK -->|"Yes"| REPLAN["🔄 Replanner: Update plan"] CHECK -->|"No"| S3["Step 3: Execute"] REPLAN --> S2B["Continue with updated plan"] S3 --> DONE["✅ Final Result"] 
# Plan and Execute — Simplified LangGraph Implementation
from typing import TypedDict, List
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o", temperature=0)  # Better planner with GPT-4

class PlanExecuteState(TypedDict):
    task: str
    plan: List[str]        # List of steps
    current_step: int      # Which step are we on
    results: List[str]     # Results of each step
    final_answer: str

def planner(state: PlanExecuteState) -> dict:
    """Task ko steps mein todo"""
    plan_prompt = f"""Break this task into 3-5 clear, executable steps:
    Task: {state['task']}
    
    Return numbered steps, one per line."""
    response = llm.invoke(plan_prompt)
    steps = [s.strip() for s in response.content.split("\n") if s.strip()]
    return {"plan": steps, "current_step": 0}

def executor(state: PlanExecuteState) -> dict:
    """Current step execute karo"""
    step = state["plan"][state["current_step"]]
    context = "\n".join(state["results"]) if state["results"] else "No prior results"
    
    exec_prompt = f"""Execute this step:
    Step: {step}
    Prior results: {context}
    
    Provide the result:"""
    response = llm.invoke(exec_prompt)
    
    new_results = state["results"] + [response.content]
    return {"results": new_results, "current_step": state["current_step"] + 1}

def should_continue(state: PlanExecuteState) -> str:
    if state["current_step"] >= len(state["plan"]):
        return "finish"
    return "execute"

def finish(state: PlanExecuteState) -> dict:
    return {"final_answer": state["results"][-1]}

# Build graph
workflow = StateGraph(PlanExecuteState)
workflow.add_node("planner", planner)
workflow.add_node("executor", executor)
workflow.add_node("finish", finish)

workflow.set_entry_point("planner")
workflow.add_edge("planner", "executor")
workflow.add_conditional_edges("executor", should_continue,
    {"execute": "executor", "finish": "finish"})
workflow.add_edge("finish", END)

app = workflow.compile()
result = app.invoke({"task": "Compare RAG vs Fine-tuning for a customer support chatbot",
    "plan": [], "current_step": 0, "results": [], "final_answer": ""})

5.5 CRAG with LangGraph — Full Implementation

Ab CRAG (Corrective RAG) ko LangGraph mein properly implement karte hain. Yeh production-ready pattern hai — retrieve karo, grade karo, agar relevant nahi toh web search se supplement karo, query transform karo agar needed ho.

flowchart TD START(["Start"]) --> RET["📚 Retrieve Documents"] RET --> GRADE["📝 Grade Documents"] GRADE --> CHECK{"All Relevant?"} CHECK -->|"Yes"| GEN["🤖 Generate Answer"] CHECK -->|"Partially"| WEB["🌐 Web Search + Docs"] CHECK -->|"No"| TRANSFORM["🔄 Transform Query"] WEB --> GEN TRANSFORM --> WEBSEARCH["🌐 Web Search (new query)"] WEBSEARCH --> GEN GEN --> DONE(["End"]) 
# CRAG — Complete LangGraph Implementation
from typing import TypedDict, List
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

class CRAGState(TypedDict):
    question: str
    documents: List[str]
    web_results: str
    generation: str
    doc_grades: List[str]  # "relevant" or "irrelevant"

# === NODE 1: Retrieve ===
def retrieve(state: CRAGState) -> dict:
    question = state["question"]
    docs = retriever.invoke(question)  # Your vector store retriever
    return {"documents": [doc.page_content for doc in docs]}

# === NODE 2: Grade Documents ===
def grade_documents(state: CRAGState) -> dict:
    question = state["question"]
    docs = state["documents"]
    grades = []
    
    grade_prompt = ChatPromptTemplate.from_template(
        """Kya yeh document user ke question ke liye relevant hai?
        Document: {document}
        Question: {question}
        Answer strictly 'yes' or 'no':"""
    )
    
    for doc in docs:
        result = llm.invoke(grade_prompt.format(document=doc, question=question))
        grades.append("relevant" if "yes" in result.content.lower() else "irrelevant")
    
    return {"doc_grades": grades}

# === NODE 3: Web Search ===
def web_search(state: CRAGState) -> dict:
    question = state["question"]
    # Use Tavily or any search API
    from langchain_community.tools.tavily_search import TavilySearchResults
    search = TavilySearchResults(max_results=3)
    results = search.invoke(question)
    web_content = "\n".join([r["content"] for r in results])
    return {"web_results": web_content}

# === NODE 4: Transform Query ===
def transform_query(state: CRAGState) -> dict:
    question = state["question"]
    transform_prompt = f"""Yeh question ka better version likho jo search mein better results laaye:
    Original: {question}
    Improved query:"""
    response = llm.invoke(transform_prompt)
    return {"question": response.content.strip()}

# === NODE 5: Generate ===
def generate(state: CRAGState) -> dict:
    question = state["question"]
    # Relevant docs + web results combine karo
    relevant_docs = [doc for doc, grade in zip(state["documents"], state["doc_grades"]) 
                     if grade == "relevant"]
    
    context = "\n".join(relevant_docs)
    if state.get("web_results"):
        context += "\n\nWeb Results:\n" + state["web_results"]
    
    gen_prompt = f"""Answer the question based on the context below.
    Context: {context}
    Question: {question}
    Answer:"""
    response = llm.invoke(gen_prompt)
    return {"generation": response.content}

# === CONDITIONAL EDGE: Decide path based on grades ===
def decide_path(state: CRAGState) -> str:
    grades = state["doc_grades"]
    relevant_count = grades.count("relevant")
    total = len(grades)
    
    if relevant_count == total:
        return "generate"           # Sab relevant — directly generate
    elif relevant_count > 0:
        return "web_search"         # Partially relevant — supplement with web
    else:
        return "transform_query"    # Nothing relevant — rewrite and search

# === BUILD GRAPH ===
crag_workflow = StateGraph(CRAGState)

crag_workflow.add_node("retrieve", retrieve)
crag_workflow.add_node("grade_documents", grade_documents)
crag_workflow.add_node("web_search", web_search)
crag_workflow.add_node("transform_query", transform_query)
crag_workflow.add_node("generate", generate)

crag_workflow.set_entry_point("retrieve")
crag_workflow.add_edge("retrieve", "grade_documents")
crag_workflow.add_conditional_edges("grade_documents", decide_path, {
    "generate": "generate",
    "web_search": "web_search",
    "transform_query": "transform_query"
})
crag_workflow.add_edge("web_search", "generate")
crag_workflow.add_edge("transform_query", "web_search")
crag_workflow.add_edge("generate", END)

crag_app = crag_workflow.compile()

# RUN!
result = crag_app.invoke({
    "question": "What are the latest advances in RAG systems?",
    "documents": [], "web_results": "", "generation": "", "doc_grades": []
})
print(result["generation"])

5.6 Self-RAG with LangGraph — Self-Correcting Agent

🧠 Ek senior engineer ki taraf se —

"Self-RAG sabse sophisticated RAG pattern hai. Agent har step pe khud ko evaluate karta hai — kya retrieval chahiye? Jo mila woh relevant hai? Mera answer faithful hai source documents ke? Answer useful hai user ke liye? Agar kisi step pe 'no' toh woh retry karta hai. Yeh pattern critical applications ke liye hai — medical, legal, financial."

# Self-RAG — Full LangGraph Implementation with Self-Correction
from typing import TypedDict, List, Optional
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

class SelfRAGState(TypedDict):
    question: str
    documents: List[str]
    generation: str
    retries: int           # Kitni baar retry kiya
    max_retries: int       # Max retries allowed

# === NODE: Route Question ===
def route_question(state: SelfRAGState) -> str:
    """Decide: kya retrieval chahiye ya direct answer de sakte hain?"""
    question = state["question"]
    route_prompt = f"""Decide if this question needs document retrieval or can be answered directly.
    - "retrieve": factual questions about specific topics
    - "direct": general opinions, greetings, simple math
    
    Question: {question}
    Decision (retrieve/direct):"""
    
    response = llm.invoke(route_prompt)
    return "retrieve" if "retrieve" in response.content.lower() else "direct"

# === NODE: Retrieve ===
def retrieve(state: SelfRAGState) -> dict:
    docs = retriever.invoke(state["question"])
    return {"documents": [doc.page_content for doc in docs]}

# === NODE: Grade Documents ===
def grade_documents(state: SelfRAGState) -> str:
    """Check: retrieved docs relevant hain?"""
    question = state["question"]
    relevant_count = 0
    
    for doc in state["documents"]:
        grade_prompt = f"Is this document relevant to '{question}'? Document: {doc[:200]}. Answer yes/no:"
        response = llm.invoke(grade_prompt)
        if "yes" in response.content.lower():
            relevant_count += 1
    
    # Agar majority relevant hai toh generate, nahi toh transform
    return "generate" if relevant_count >= len(state["documents"]) // 2 else "transform"

# === NODE: Generate ===
def generate(state: SelfRAGState) -> dict:
    context = "\n".join(state["documents"])
    gen_prompt = f"Based ONLY on this context, answer the question.\nContext: {context}\nQuestion: {state['question']}\nAnswer:"
    response = llm.invoke(gen_prompt)
    return {"generation": response.content}

# === NODE: Grade Generation ===
def grade_generation(state: SelfRAGState) -> str:
    """Check: answer faithful hai documents ke? Aur useful hai?"""
    # Check faithfulness (ISSUP)
    faith_prompt = f"""Is this answer supported by the documents?
    Documents: {' '.join(state['documents'][:2])}
    Answer: {state['generation']}
    Supported (yes/no):"""
    faith_response = llm.invoke(faith_prompt)
    
    if "no" in faith_response.content.lower():
        return "not_faithful"  # Hallucinating! Retry.
    
    # Check usefulness (ISUSE) 
    useful_prompt = f"""Does this answer actually help with the question?
    Question: {state['question']}
    Answer: {state['generation']}
    Useful (yes/no):"""
    useful_response = llm.invoke(useful_prompt)
    
    if "no" in useful_response.content.lower():
        return "not_useful"  # Not helpful! Transform and retry.
    
    return "useful"  # All good!

# === NODE: Transform Query ===
def transform_query(state: SelfRAGState) -> dict:
    transform_prompt = f"Rewrite this query for better search results: {state['question']}"
    response = llm.invoke(transform_prompt)
    return {"question": response.content.strip(), "retries": state["retries"] + 1}

# === NODE: Direct Generate ===
def generate_direct(state: SelfRAGState) -> dict:
    response = llm.invoke(state["question"])
    return {"generation": response.content}

# === CONDITIONAL: After grading generation ===
def after_grade_generation(state: SelfRAGState) -> str:
    grade = grade_generation(state)
    if grade == "useful":
        return "end"
    elif state["retries"] >= state.get("max_retries", 3):
        return "end"  # Max retries reached, give what we have
    else:
        return "transform"

# === BUILD SELF-RAG GRAPH ===
self_rag = StateGraph(SelfRAGState)

self_rag.add_node("retrieve", retrieve)
self_rag.add_node("generate", generate)
self_rag.add_node("generate_direct", generate_direct)
self_rag.add_node("transform_query", transform_query)

# Entry point based on routing
self_rag.set_conditional_entry_point(route_question, {
    "retrieve": "retrieve",
    "direct": "generate_direct"
})

# After retrieve -> grade docs -> generate or transform
self_rag.add_conditional_edges("retrieve", grade_documents, {
    "generate": "generate",
    "transform": "transform_query"
})

# After generate -> grade answer -> end or transform
self_rag.add_conditional_edges("generate", after_grade_generation, {
    "end": END,
    "transform": "transform_query"
})

self_rag.add_edge("generate_direct", END)
self_rag.add_edge("transform_query", "retrieve")  # Retry loop!

self_rag_app = self_rag.compile()

# Run with self-correction!
result = self_rag_app.invoke({
    "question": "What is our company's remote work policy?",
    "documents": [], "generation": "", "retries": 0, "max_retries": 3
})
