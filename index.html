Section 1

RAG Fundamentals

Retrieval-Augmented Generation ko zero se samjho — problem se shuru, solution tak.

1 RAG Kya Hai — Problem Se Shuru Karte Hain

👨‍💻 Ek senior engineer ki taraf se —

Production mein 90% LLM failures ka reason ek hi hai: model ko relevant context nahi mila. RAG usi problem ka solution hai.

Phase 1 — The Problem

flowchart TD
    A[LLM Training Complete] -->|Knowledge Frozen| B[Training Data Cutoff]
    B --> C{User Asks New/Private Question}
    C -->|No Data Available| D[Hallucination 💀]
    C -->|Honest Mode| E[I Don't Know]
    style A fill:#6366f1,color:#fff
    style D fill:#ef4444,color:#fff
    style E fill:#f59e0b,color:#fff

LLM ki knowledge training ke time pe freeze ho jaati hai. Agar September 2024 tak train hua, toh uske baad ki koi cheez usse pata nahi.

Har cheez ke liye dobara train karna weeks of compute lagta hai — impractical aur expensive.

Private company data toh LLM ne kabhi dekha hi nahi — tumhari internal docs, customer data, proprietary knowledge — sab invisible hai.

Phase 2 — The Solution: RAG

flowchart LR
    A[User Query] --> B[Embed Query]
    B --> C[Vector DB Search]
    C --> D[Top-K Chunks Retrieved]
    D --> E[Query + Chunks = Prompt]
    E --> F[LLM]
    F --> G[Grounded Answer ✅]
    style A fill:#6366f1,color:#fff
    style C fill:#f59e0b,color:#fff
    style F fill:#6366f1,color:#fff
    style G fill:#22c55e,color:#fff

User Query → User apna question puchta hai normally.

Embed Query → Question ko vector (numbers) mein convert karo.

Vector DB Search → Similar vectors dhundho database mein — yeh relevant documents deta hai.

Top-K Chunks → Sabse relevant k chunks retrieve karo (k=3 to 5 usually).

Query + Chunks = Prompt → Original question + retrieved context = final prompt for LLM.

LLM → Ab LLM ke paas relevant info hai, hallucinate karne ki zaroorat nahi.

Grounded Answer → Answer based on real data, not imagination.

💡 Analogy

Tu exam dene ja raha hai open-book ke saath. Vector DB teri reference book hai. LLM tera dimaag hai jo us book ko padhke answer likhta hai. Without the book? Hallucination.

Phase 3 — Why It Works

LLM ko relevant context already prompt mein mil jaata hai. Usse guess karne ki zaroorat nahi — directly context se answer generate karta hai. Simple but powerful.

Working Code — End-to-End RAG Pipeline

# RAG Pipeline - Complete Example
# Ek senior engineer ki taraf se: yeh production ka starting point hai

from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import Chroma
from langchain.chains import RetrievalQA

# Step 1: Document load karo (PDF se text nikalo)
loader = PyPDFLoader("company_docs.pdf")
documents = loader.load()  # Poori PDF pages mein split ho jaati hai

# Step 2: Chunks mein todo (bade documents ko chote pieces mein)
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # Har chunk max 500 characters
    chunk_overlap=50,     # 50 chars overlap — context boundary pe loss na ho
    separators=["\n\n", "\n", ". ", " ", ""]  # Pehle paragraphs pe split karo
)
chunks = text_splitter.split_documents(documents)

# Step 3: Embeddings banao (text ko numbers mein convert karo)
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# Step 4: Vector store mein daalo (Chroma local DB use kar rahe)
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"  # Local disk pe save hoga
)

# Step 5: Retriever banao (search interface)
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 3}  # Top 3 relevant chunks laao
)

# Step 6: QA Chain banao (retriever + LLM connect karo)
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",  # Saare chunks ek prompt mein daal do
    retriever=retriever,
    return_source_documents=True  # Sources bhi return karo
)

# Step 7: Question pucho!
result = qa_chain.invoke({"query": "Company ki leave policy kya hai?"})
print(result["result"])  # Grounded answer from your docs

⚠️ Common Mistakes

  • Chunk size too small → context missing → bad answers
  • Different embedding model for docs vs queries → results garbage
  • k=1 → almost always wrong. Minimum k=3 use karo.
  • No metadata stored → source tracking impossible later

# Chunking Strategies - Sabse Common Approaches

from langchain.text_splitter import (
    RecursiveCharacterTextSplitter,
    CharacterTextSplitter,
    TokenTextSplitter
)
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

# 1. Recursive (DEFAULT - almost always use this)
recursive_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,        # Max characters per chunk
    chunk_overlap=50,      # Overlap — so context boundary pe info na lose ho
    separators=["\n\n", "\n", ". ", " ", ""]  # Pehle natural breaks try karo
)

# 2. Token-based (jab token budget tight ho)
token_splitter = TokenTextSplitter(
    chunk_size=256,        # Tokens, characters nahi
    chunk_overlap=20
)

# 3. Semantic (jab meaning matters most — slow but accurate)
semantic_splitter = SemanticChunker(
    OpenAIEmbeddings(),
    breakpoint_threshold_type="percentile"  # Auto-detect meaning breaks
)

# Usage
text = "Your long document text here..."
chunks = recursive_splitter.split_text(text)
print(f"Total chunks: {len(chunks)}")
print(f"First chunk: {chunks[0][:100]}...")

⚠️ Common Mistakes

  • Chunk overlap bhool gaye → answers cut at context boundary
  • chunk_size bahut bada → embedding quality drops (meaning diluted)
  • chunk_size aur embedding model's max token limit align nahi kiya → truncation

🔑 Key Insight

embed_documents() vs embed_query() — always use the right method. Some models optimize them differently. Documents ke liye batch embedding use karo, query ke liye single embedding.

# Embeddings - OpenAI + Ollama (Local)

from langchain_openai import OpenAIEmbeddings
from langchain_community.embeddings import OllamaEmbeddings

# OpenAI Embeddings (cloud, paid, best quality)
openai_embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",  # Best value: cheap + accurate
    # model="text-embedding-3-large",  # Jab accuracy critical ho
)

# Ollama Embeddings (local, free, private)
ollama_embeddings = OllamaEmbeddings(
    model="nomic-embed-text",  # Free, runs locally
    base_url="http://localhost:11434"  # Ollama server
)

# Single text embed karo (query ke liye)
query_vector = openai_embeddings.embed_query("What is RAG?")

# Multiple documents embed karo (batch - faster + cheaper)
docs = ["RAG uses retrieval", "Agents use tools", "LLMs generate text"]
doc_vectors = openai_embeddings.embed_documents(docs)
print(f"Embedded {len(doc_vectors)} documents")

5 Retrieval Techniques — Sahi Chunks Kaise Laayein

1. Basic Similarity Search (k-NN)

flowchart LR
    A[Query Vector] --> B[Compare with ALL vectors]
    B --> C[Sort by distance]
    C --> D[Return Top-K closest]
    style A fill:#6366f1,color:#fff
    style D fill:#22c55e,color:#fff

Sabse simple — query vector ko saare stored vectors se compare karo, closest k laao.

k tuning: k=1 bad (too narrow), k=3-5 good (balanced), k=20 for reranking pipeline.

2. Score Threshold

flowchart LR
    A[Retrieved Chunks] --> B{Score > 0.75?}
    B -->|Yes| C[Keep - Relevant ✅]
    B -->|No| D[Discard - Garbage ❌]
    style B fill:#f59e0b,color:#fff
    style C fill:#22c55e,color:#fff
    style D fill:#ef4444,color:#fff

Problem: Retriever ALWAYS returns k results, even when no relevant doc exists — garbage in, garbage out.

Solution: Minimum score threshold 0.7-0.8 set karo. Below threshold = no results (better than bad results).

3. MMR (Maximal Marginal Relevance)

flowchart TD
    A[Fetch top-20 candidates] --> B[Pick most relevant as #1]
    B --> C[For each remaining:]
    C --> D{Relevant to query AND Different from already picked?}
    D -->|Both| E[Add to results]
    D -->|Too similar to picked| F[Skip]
    E --> G[Final diverse Top-5]
    style A fill:#6366f1,color:#fff
    style D fill:#f59e0b,color:#fff
    style G fill:#22c55e,color:#fff

Problem: 5 results mein 4 almost identical chunks — redundancy, no new info.

Solution: Relevance + Diversity balance. lambda_mult: 1.0 = pure relevance, 0.0 = max diversity. Sweet spot: 0.5-0.7.

fetch_k=20 (broad candidates), then MMR narrows to k=5 diverse results.

4. Hybrid Search (Dense + Sparse)

flowchart TD
    A[User Query] --> B[Dense: Semantic Vectors]
    A --> C[Sparse: BM25 Keywords]
    B --> D[Semantic Results]
    C --> E[Keyword Results]
    D --> F[EnsembleRetriever]
    E --> F
    F --> G[Merged + Deduplicated]
    style A fill:#6366f1,color:#fff
    style F fill:#f59e0b,color:#fff
    style G fill:#22c55e,color:#fff

Problem: Pure semantic search misses exact terms — "React hooks v18.2" ke liye semantic similarity kaam nahi karti.

Solution: Dense (vectors) + Sparse (BM25 keyword matching) combine karo. Alpha parameter controls weight: 0.7 semantic + 0.3 keyword is typical.

5. Contextual Compression

flowchart LR
    A[500 token chunk] --> B[Compressor]
    B --> C[Only 2 relevant sentences kept]
    C --> D[Compressed context to LLM]
    style A fill:#ef4444,color:#fff
    style B fill:#f59e0b,color:#fff
    style C fill:#22c55e,color:#fff

Problem: 500-token chunk mein sirf 2 sentences actually relevant hain — baaqi noise.

LLMChainExtractor: Most accurate, uses LLM to extract. EmbeddingsFilter: Faster, uses similarity threshold. Trade-off: accuracy vs speed.

6. Parent Document Retriever

flowchart TD
    A[Large Parent Doc 2000 tokens] --> B[Split into Child Chunks 200 tokens each]
    B --> C[Index CHILDREN in Vector DB]
    C --> D[Query matches Child Chunk #3]
    D --> E[Return PARENT document]
    E --> F[LLM gets full context]
    style C fill:#6366f1,color:#fff
    style D fill:#f59e0b,color:#fff
    style E fill:#22c55e,color:#fff

Dilemma: Small chunks = better retrieval precision. Large chunks = better generation quality. Can't have both?

Solution: Index small (retrieve accurately), but RETURN large parent (generate with full context). Best of both worlds.

7. Multi-Query Retriever

flowchart TD
    A[Original Query] --> B[LLM generates 3-5 variations]
    B --> C[Query 1: Rephrased]
    B --> D[Query 2: Synonym version]
    B --> E[Query 3: Different angle]
    C --> F[Retrieve for each]
    D --> F
    E --> F
    F --> G[Deduplicate + Merge]
    style A fill:#6366f1,color:#fff
    style B fill:#f59e0b,color:#fff
    style G fill:#22c55e,color:#fff

Problem: Tumhara exact phrasing relevant docs miss kar sakta hai. Solution: LLM se 3-5 different phrasings generate karao, har ek ke liye retrieve karo, merge karo. Recall significantly improves.

8. Re-Ranking (Two-Stage Retrieval)

flowchart LR
    A[Query] --> B[Stage 1: Bi-Encoder]
    B --> C[Fast: Top-20 candidates]
    C --> D[Stage 2: Cross-Encoder]
    D --> E[Accurate: Top-5 reranked]
    style B fill:#6366f1,color:#fff
    style D fill:#f59e0b,color:#fff
    style E fill:#22c55e,color:#fff

Stage 1 (Bi-Encoder): Fast but approximate — retrieve top-20. Stage 2 (Cross-Encoder): Slow but accurate — rerank to top-5. Cross-encoder sees query+doc together, so much better at judging relevance.

# Advanced Retrieval Techniques - All in One

from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.retrievers import (
    ContextualCompressionRetriever,
    MultiQueryRetriever,
    ParentDocumentRetriever
)
from langchain.retrievers.document_compressors import LLMChainExtractor
from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma(persist_directory="./db", embedding_function=embeddings)

# 1. MMR Search (diversity + relevance)
mmr_results = vectorstore.max_marginal_relevance_search(
    "What is chunking?",
    k=5,              # Final results chahiye
    fetch_k=20,       # Broad candidates fetch karo
    lambda_mult=0.6   # 0.6 = slightly favor relevance over diversity
)

# 2. Score Threshold (garbage filter)
retriever = vectorstore.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={"score_threshold": 0.75, "k": 5}
)

# 3. Hybrid Search (semantic + keyword)
bm25_retriever = BM25Retriever.from_documents(docs, k=5)
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 5})
hybrid = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.3, 0.7]  # 70% semantic, 30% keyword
)

# 4. Multi-Query (multiple phrasings)
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.3)
multi_query = MultiQueryRetriever.from_llm(
    retriever=vector_retriever, llm=llm
)

# 5. Contextual Compression (extract only relevant parts)
compressor = LLMChainExtractor.from_llm(llm)
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=vector_retriever
)

🧠
LLM (Brain)

Decision-making core. Reasoning, understanding, generating. Without it: no intelligence.

🛠️
Tools (Hands)

External actions — search, code execution, API calls. Without it: can only talk, not do.

💾
Memory (Storage)

Conversation history, learned facts. Without it: goldfish memory, repeats work.

📋
Planning (Strategy)

🔄
Observation Loop

Check results, adapt, retry. Without it: blind execution, no self-correction.

❌ Common Misconception

Wrong: "Agent = calling a tool"

Right: Agent = LLM that decides WHICH tool, evaluates the result, decides WHAT NEXT — in a loop. The loop + decision-making is what makes it an agent.

Code Review

8 Agent Architecture — Andar Se Kaise Kaam Karta Hai

    D --> H[API Caller]
    E --> I[Observation]
    F --> I
    G --> I
    H --> I
    I --> J[Memory Update]
    J --> B
    B --> K{Task Complete?}

    style I fill:#22d3ee,color:#000
    style L fill:#22c55e,color:#fff

User Input: Natural language instruction — can be simple or complex multi-step task.

LLM Brain: The core reasoning engine. Interprets input, maintains context, makes decisions at every step. Without it, nothing works.

Planning Module: Decomposes complex tasks into executable steps. Prioritizes, sequences, and manages dependencies between steps.

Tool Selection: Agent decides WHICH tool to call based on current step's needs. This is the key decision point — wrong tool = wasted computation.

Observation: Tool results come back as "observations". Agent reads them, evaluates quality, decides if more action needed.

Memory Update: New information stored — conversation context, tool results, intermediate findings. Prevents redundant work.

Loop Decision: After each observation, agent asks: "Do I have enough to answer?" If yes → final answer. If no → back to planning.

@tool
def calculate_token_cost(text: str, model: str = "gpt-4o-mini") -> str:
    """Calculate approximate token cost for given text and model.
    Use this when user asks about cost estimation."""
    # Approximate: 1 token ~ 4 characters
    tokens = len(text) / 4
    costs = {"gpt-4o-mini": 0.00015, "gpt-4o": 0.005}
    cost = (tokens / 1000) * costs.get(model, 0.001)
    return f"~{int(tokens)} tokens, cost: ${cost:.4f}"

11 LangGraph Kya Hai — Why Simple Chains Aren't Enough

👨‍💻 Ek senior engineer ki taraf se —

LangChain simple chains ke liye theek hai. But jab tumhe loops chahiye, conditional branching chahiye, state management chahiye — tab LangGraph use karo. It's the difference between a script and a program.

Simple Chain ❌

flowchart LR
    A[A] --> B[B]
    B --> C[C]
    C --> D[END]
    style A fill:#ef4444,color:#fff
    style D fill:#ef4444,color:#fff

Linear. No loops. No conditions. Limited.

LangGraph ✅

flowchart LR
    A[A] --> B[B]
    B --> C{Decision}
    C -->|Path 1| D[D]
    C -->|Path 2| B

Loops, branches, conditional logic. Real programs.

Core Concepts

State

TypedDict shared dictionary — har node isko read/write karta hai.

class AgentState(TypedDict):
    messages: list
    context: str
    should_continue: bool
Nodes

Functions jo state lete hain, ek kaam karte hain, updated state return karte hain.

    return {"context": docs}
Normal Edges

Always A → B. Unconditional flow. Simple connections between nodes.

Conditional Edges

Function decides which node next. This is where the magic happens — branching logic.

# LangGraph Basics - StateGraph Construction

from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, END
from langgraph.graph.message import add_messages

# Step 1: State define karo (shared data structure)
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]  # Message list (auto-append)
    context: str                              # Retrieved documents
    should_continue: bool                     # Loop control flag

# Step 2: Nodes define karo (har node ek function hai)
def retrieve_node(state: AgentState):
    """Documents retrieve karo based on latest message"""
    query = state["messages"][-1].content
    docs = retriever.invoke(query)
    return {"context": "\n".join([d.page_content for d in docs])}

def generate_node(state: AgentState):
    """LLM se answer generate karo"""
    prompt = f"Context: {state['context']}\n\nQuestion: {state['messages'][-1].content}"
    response = llm.invoke(prompt)
    return {"messages": [response]}

# Step 3: Conditional edge function
def should_retrieve(state: AgentState) -> str:
    """Decide karo: retrieve karna hai ya directly answer karna hai"""
    last_msg = state["messages"][-1].content
    if len(last_msg) < 10:  # Simple greeting
        return "generate"
    return "retrieve"

# Step 4: Graph build karo
graph = StateGraph(AgentState)
graph.add_node("retrieve", retrieve_node)
graph.add_node("generate", generate_node)

# Edges add karo
graph.set_entry_point("retrieve")  # Start yahaan se
graph.add_edge("retrieve", "generate")  # retrieve → generate always
graph.add_edge("generate", END)  # generate → END

# Compile karo (ab runnable hai)
app = graph.compile()

# Run karo!
result = app.invoke({"messages": [("human", "RAG kya hai?")], "context": "", "should_continue": True})

# ReAct Pattern with LangGraph - Full Implementation

from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI
from langchain_community.tools.tavily_search import TavilySearchResults

# Tools define karo
search = TavilySearchResults(max_results=3)
tools = [search]

# LLM with tool binding
llm = ChatOpenAI(model="gpt-4o-mini").bind_tools(tools)

# State
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]

# Node: LLM call karo (reasoning step)
def call_model(state: AgentState):
    """LLM ko current state do, wo decide karega tool call ya final answer"""
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

# Conditional: aage tool call karna hai ya answer dena hai?
def should_continue(state: AgentState) -> str:
    """Check karo: LLM ne tool call kiya ya final answer diya?"""
    last_message = state["messages"][-1]
    # Agar tool_calls hai toh tools run karo
    if last_message.tool_calls:
        return "tools"
    # Nahi toh answer de do
    return END

# Graph build karo
graph = StateGraph(AgentState)

# Nodes add karo
graph.add_node("agent", call_model)          # LLM reasoning
graph.add_node("tools", ToolNode(tools))     # Tool execution

# Entry point
graph.set_entry_point("agent")

# Conditional edge: agent ke baad decide karo
graph.add_conditional_edges("agent", should_continue, {"tools": "tools", END: END})

# Tools ke baad wapas agent pe jaao (loop!)
graph.add_edge("tools", "agent")

# Compile
react_agent = graph.compile()

# Run karo!
result = react_agent.invoke({
    "messages": [("human", "What are the latest developments in RAG?")]
})
print(result["messages"][-1].content)

One step at a time. No foresight. Good for simple tasks but inefficient for complex ones.

# Plan and Execute Pattern with LangGraph

from typing import TypedDict, List
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")

class PlanExecuteState(TypedDict):
    input: str                # Original task
    plan: List[str]           # Steps list
    current_step: int         # Which step we're on
    results: List[str]        # Results from each step
    final_answer: str         # Final synthesized answer

def planner(state: PlanExecuteState):
    """Task ko steps mein todo"""
    prompt = f"Break this task into 3-5 clear steps:\n{state['input']}"
    response = llm.invoke(prompt)
    steps = response.content.split("\n")
    return {"plan": steps, "current_step": 0, "results": []}

def executor(state: PlanExecuteState):
    """Current step execute karo"""
    step = state["plan"][state["current_step"]]
    prompt = f"Execute this step: {step}\nPrevious results: {state['results']}"
    response = llm.invoke(prompt)
    results = state["results"] + [response.content]
    return {"results": results, "current_step": state["current_step"] + 1}

def synthesizer(state: PlanExecuteState):
    """Saare results combine karke final answer do"""
    prompt = f"Synthesize final answer from:\n{chr(10).join(state['results'])}"
    response = llm.invoke(prompt)
    return {"final_answer": response.content}

def should_continue(state: PlanExecuteState) -> str:
    if state["current_step"] >= len(state["plan"]):
        return "synthesize"
    return "execute"

# Graph
graph = StateGraph(PlanExecuteState)
graph.add_node("plan", planner)
graph.add_node("execute", executor)
graph.add_node("synthesize", synthesizer)

graph.set_entry_point("plan")
graph.add_edge("plan", "execute")
graph.add_conditional_edges("execute", should_continue, {"execute": "execute", "synthesize": "synthesize"})
graph.add_edge("synthesize", END)

plan_execute_agent = graph.compile()

# Multi-Agent Supervisor Pattern

from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, END
from langgraph.graph.message import add_messages
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")

class SupervisorState(TypedDict):
    messages: Annotated[list, add_messages]
    next_agent: str

# Worker Agents
def research_agent(state: SupervisorState):
    """Web search karke information laao"""
    prompt = "You are a research agent. Search and summarize info for: " + state["messages"][-1].content
    response = llm.invoke(prompt)
    return {"messages": [("assistant", f"[Research]: {response.content}")]}

def code_agent(state: SupervisorState):
    """Code likhne/explain karne ka kaam"""
    prompt = "You are a coding agent. Write/explain code for: " + state["messages"][-1].content
    response = llm.invoke(prompt)
    return {"messages": [("assistant", f"[Code]: {response.content}")]}

def rag_agent(state: SupervisorState):
    """Internal documents se answer laao"""
    prompt = "You are a RAG agent. Retrieve and answer from docs: " + state["messages"][-1].content
    response = llm.invoke(prompt)
    return {"messages": [("assistant", f"[RAG]: {response.content}")]}

# Supervisor: decide karo kaunsa agent call karna hai
def supervisor(state: SupervisorState):
    """Decide which worker agent to route to next"""
    prompt = f"""You are a supervisor. Based on the conversation, decide next action.
Options: 'research', 'code', 'rag', 'FINISH'
Conversation: {state['messages'][-1].content}
Respond with ONLY one word."""
    response = llm.invoke(prompt)
    return {"next_agent": response.content.strip().lower()}

def route_supervisor(state: SupervisorState) -> str:
    next_agent = state.get("next_agent", "finish")
    if "research" in next_agent: return "research"
    if "code" in next_agent: return "code"
    if "rag" in next_agent: return "rag"
    return END

# Build graph
graph = StateGraph(SupervisorState)
graph.add_node("supervisor", supervisor)
graph.add_node("research", research_agent)
graph.add_node("code", code_agent)
graph.add_node("rag", rag_agent)

graph.set_entry_point("supervisor")
graph.add_conditional_edges("supervisor", route_supervisor,
    {"research": "research", "code": "code", "rag": "rag", END: END})
graph.add_edge("research", "supervisor")
graph.add_edge("code", "supervisor")
graph.add_edge("rag", "supervisor")

multi_agent = graph.compile()

# Corrective RAG (CRAG) - Full LangGraph Implementation

from typing import TypedDict, List
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI
from langchain_community.tools.tavily_search import TavilySearchResults

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
web_search = TavilySearchResults(max_results=3)

class CRAGState(TypedDict):
    question: str
    documents: List[str]
    web_results: List[str]
    generation: str
    relevance_scores: List[str]  # "relevant" or "irrelevant"

def retrieve(state: CRAGState):
    """Step 1: Documents retrieve karo"""
    docs = retriever.invoke(state["question"])
    return {"documents": [d.page_content for d in docs]}

def grade_documents(state: CRAGState):
    """Step 2: Har document ko grade karo - relevant ya nahi?"""
    scores = []
    for doc in state["documents"]:
        prompt = f"Is this document relevant to '{state['question']}'?\nDoc: {doc[:200]}\nAnswer 'relevant' or 'irrelevant':"
        score = llm.invoke(prompt).content.strip().lower()
        scores.append(score)
    return {"relevance_scores": scores}

def decide_path(state: CRAGState) -> str:
    """Conditional: kitne docs relevant hain?"""
    relevant = sum(1 for s in state["relevance_scores"] if "relevant" in s)
    if relevant >= 2: return "generate"      # Enough good docs
    return "web_search"                       # Need web backup

def transform_query(state: CRAGState):
    """Query rewrite karo better results ke liye"""
    prompt = f"Rewrite this query for better search results: {state['question']}"
    better_query = llm.invoke(prompt).content
    return {"question": better_query}

def web_search_node(state: CRAGState):
    """Web se additional info laao"""
    results = web_search.invoke(state["question"])
    web_docs = [r["content"] for r in results] if results else []
    return {"web_results": web_docs}

def generate(state: CRAGState):
    """Final answer generate karo"""
    context = "\n".join(state["documents"] + state.get("web_results", []))
    prompt = f"Answer based on context:\nContext: {context}\nQuestion: {state['question']}"
    answer = llm.invoke(prompt).content
    return {"generation": answer}

# Build CRAG Graph
graph = StateGraph(CRAGState)
graph.add_node("retrieve", retrieve)
graph.add_node("grade", grade_documents)
graph.add_node("transform", transform_query)
graph.add_node("web_search", web_search_node)
graph.add_node("generate", generate)

graph.set_entry_point("retrieve")
graph.add_edge("retrieve", "grade")
graph.add_conditional_edges("grade", decide_path, {"generate": "generate", "web_search": "transform"})
graph.add_edge("transform", "web_search")
graph.add_edge("web_search", "generate")
graph.add_edge("generate", END)

crag_app = graph.compile()

class SelfRAGState(TypedDict):

    documents: List[str]

def route_question(state: SelfRAGState) -> str:
    """Decide: retrieval needed ya direct answer de sakte hain?"""
    prompt = f"Does this need retrieval or can be answered directly? '{state['question']}'\nAnswer: 'retrieve' or 'direct'"
    response = llm.invoke(prompt).content.strip().lower()
    return "retrieve" if "retrieve" in response else "generate"

def retrieve(state: SelfRAGState):
    docs = retriever.invoke(state["question"])
    return {"documents": [d.page_content for d in docs]}

def grade_documents(state: SelfRAGState) -> str:
    """Grade: docs relevant hain?"""
    relevant = 0
    for doc in state["documents"][:3]:
        prompt = f"Relevant? Q: {state['question']}\nDoc: {doc[:150]}\nAnswer yes/no:"
        if "yes" in llm.invoke(prompt).content.lower():
            relevant += 1
    return "generate" if relevant >= 1 else "transform"

def generate(state: SelfRAGState):
    context = "\n".join(state["documents"]) if state["documents"] else ""
    prompt = f"Context: {context}\nQuestion: {state['question']}\nAnswer:"
    answer = llm.invoke(prompt).content
    return {"generation": answer}

def grade_generation(state: SelfRAGState) -> str:
    """Self-check: answer supported + useful?"""
    if state.get("retries", 0) >= 3:
        return END  # Max retries hit
    check = f"Is this answer supported by docs and useful?\nAnswer: {state['generation']}\nDocs: {str(state['documents'])[:300]}\nRespond: 'good' or 'bad'"
    result = llm.invoke(check).content.lower()
    return END if "good" in result else "transform"

def transform_query(state: SelfRAGState):
    prompt = f"Rewrite for better results: {state['question']}"
    new_q = llm.invoke(prompt).content
    return {"question": new_q, "retries": state.get("retries", 0) + 1}

# Build Self-RAG Graph
graph = StateGraph(SelfRAGState)
graph.add_node("retrieve", retrieve)
graph.add_node("generate", generate)
graph.add_node("transform", transform_query)

graph.set_conditional_entry_point(route_question, {"retrieve": "retrieve", "generate": "generate"})
graph.add_conditional_edges("retrieve", grade_documents, {"generate": "generate", "transform": "transform"})
graph.add_conditional_edges("generate", grade_generation, {END: END, "transform": "transform"})
graph.add_edge("transform", "retrieve")

self_rag_app = graph.compile()

Prompt Injection

Malicious doc text overrides system instructions. Defense: never put retrieved text in system prompt position.

Data Leakage

Restricted chunks returned to wrong user. Defense: metadata filters per user/role at retrieval time.

Jailbreaks

Crafted queries bypass system prompt. Defense: output filtering + guardrails (NeMo Guardrails).

Indirect Injection

Attacker embeds instructions in documents you'll retrieve. Defense: content sanitization + separate context/instruction sections.

6.5 Cost Optimization

Cost Breakdown per RAG Request
✅ Optimization Strategies
  • Semantic Caching: Skip re-embedding similar queries (GPTCache)
  • Adaptive RAG: Skip retrieval for simple questions
  • Smaller Models: Use 3-small for non-critical embeddings
  • Batch Embeddings: Bulk process during indexing
❌ Cost Killers
  • Large k values: k=20 without reranking = wasted tokens
  • No caching: Same query hits DB + LLM every time
  • GPT-4 for everything: Use mini for grading, full for generation
  • No monitoring: Can't optimize what you don't measure

💰 Production Reality

Most production failures aren't technical — they're financial. Always calculate cost per query before launch. A system that costs $0.50/query with 10K daily users = $5,000/day = $150K/month.

6.7 Long Context Models vs RAG

Factor Long Context LLM RAG
CostVery high (all tokens processed)Lower, targeted retrieval
LatencySlow (large context = slow)Fast (small, focused context)
AccuracyDegrades ("lost in the middle")Focused on relevant chunks
Best ForReasoning over small/medium docsLarge knowledge bases
MaintenanceNone (just feed doc)Index needs updates

💡 "Lost in the Middle" Problem

Research shows LLMs pay most attention to the beginning and end of context. Information in the middle gets "lost". This is why stuffing everything into a long context window isn't always better than targeted RAG retrieval.

Conclusion: Most production systems combine both — RAG for retrieval efficiency (find the right 5 chunks out of 1M), longer context for synthesis (reason over those 5 chunks together).
