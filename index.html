2. RAG Fundamentals — Deep Dive 📚

Core architecture samjho — Google-level depth mein.

2.1 RAG Kya Hai? 🤔

RAG (Retrieval-Augmented Generation) = LLM ko generation se pehle external knowledge base se relevant context dena. Problems solved: Hallucination reduction, Knowledge cutoff bypass, Source attribution, Domain injection without fine-tuning.

Google Interview: "RAG is non-parametric memory augmentation — LLM's parametric knowledge (weights) supplemented by non-parametric knowledge (retrieved docs). Open-book exam vs closed-book."

End-to-End Data Flow

graph LR A[User Query] --> B[Embedding Model] B --> C[Query Vector] C --> D[Vector DB Search] D --> E[Top-K Documents] E --> F[Prompt + Context] F --> G[LLM] G --> H[Answer]

2.4 Embeddings Deep Dive 🔢

Text → dense vectors. Similar meaning = geometrically close. Distance metrics: Cosine (angle, most common), Euclidean (L2 distance), Dot Product (direction + magnitude).

Cosine: Ignores magnitude. Best for text (document length varies). Range -1 to 1.

Euclidean: Straight-line distance. Good for normalized vectors only.

Dot Product: Direction + magnitude. Fastest. Equals cosine if pre-normalized.

Google Interview: "If you normalize all vectors to unit length, dot product = cosine and is computationally cheaper. That's why production systems normalize at index time."

3. Advanced RAG ⚡

Production patterns jo basic RAG se 30-40% better results dete hain.

3.1 Retrieval Techniques

How: Query embed → cosine similarity vs all stored vectors → top-k. Complexity: Brute O(n), HNSW O(log n). Limitation: Redundant results, no diversity.

results = db.similarity_search_with_relevance_scores("query", k=4)
filtered = [(doc, s) for doc, s in results if s > 0.75]

retriever = db.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={"score_threshold": 0.75, "k": 6})

MMR = relevance + diversity. Formula: lambda * sim(doc,query) - (1-lambda) * max(sim(doc,selected)). Lambda 1=relevance, 0=diversity. Best: 0.5-0.7.

retriever = db.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 4, "fetch_k": 20, "lambda_mult": 0.6}
)
# fetch_k=20: candidate pool. k=4: final results. O(k*fetch_k)

Hybrid = Dense (semantic) + Sparse (BM25 keyword). Dense catches synonyms, sparse catches exact terms. Combined best for technical/legal docs.

graph LR Q[Query] --> D[Dense Embeddings] Q --> S[Sparse BM25] D --> E[EnsembleRetriever] S --> E E --> R[Ranked Results] 
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever
bm25 = BM25Retriever.from_documents(chunks, k=6)
semantic = db.as_retriever(search_kwargs={"k": 6})
hybrid = EnsembleRetriever(retrievers=[bm25, semantic], weights=[0.4, 0.6])

Contextual Compression: Retrieved chunks mein noise hota hai. Compression sirf relevant part extract karta hai → less tokens → better answers.

from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor, EmbeddingsFilter

# Method 1: LLM-based (best quality, expensive)
compressor = LLMChainExtractor.from_llm(llm)
# Method 2: Embedding filter (fast, cheap)
emb_filter = EmbeddingsFilter(embeddings=emb, similarity_threshold=0.8)

cc = ContextualCompressionRetriever(base_compressor=compressor, base_retriever=retriever)

3.2 Advanced Patterns

RAG Fusion 🔀

Multiple query variations generate → each searches VectorDB → Reciprocal Rank Fusion combines. Recall 20-30% improve hota hai.

graph LR Q[Original Query] --> LLM[LLM generates 3 variants] LLM --> V1[Search variant 1] LLM --> V2[Search variant 2] LLM --> V3[Search variant 3] V1 --> RRF[Reciprocal Rank Fusion] V2 --> RRF V3 --> RRF RRF --> A[Final Ranked Docs] 
# Reciprocal Rank Fusion scoring
def rrf(results_list, k=60):
    scores = {}
    for results in results_list:
        for rank, doc in enumerate(results):
            scores[doc.page_content] = scores.get(doc.page_content, 0) + 1/(k + rank)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)

HyDE — Hypothetical Document Embeddings 💭

Query ko directly embed karne ki jagah, pehle LLM se hypothetical answer generate karao, usse embed karke search karo. Query-document semantic gap bridge karta hai.

graph LR Q[Query] --> H[LLM: Hypothetical Answer] H --> E[Embed hypothetical] E --> V[VectorDB Search] V --> R[Real Documents] R --> G[Generate Real Answer] 
from langchain.chains import HypotheticalDocumentEmbedder
hyde = HypotheticalDocumentEmbedder.from_llm(llm, base_embeddings, prompt_key="web_search")
# Now hyde.embed_query() generates hypothetical doc first, then embeds it

CRAG — Corrective RAG 🩹

Retrieved docs ko grade karo (relevant/not). Not relevant? Web search fallback + query rewrite. Production mein retrieval failure gracefully handle karta hai.

graph LR Q[Query] --> R[Retrieve Docs] R --> G{Grade Relevance} G -->|Relevant| Gen[Generate Answer] G -->|Not Relevant| W[Web Search Fallback] W --> Gen 
def grade_documents(state):
    docs = state["documents"]
    filtered = []
    for doc in docs:
        score = grader_llm.invoke(f"Is this relevant to: {state['question']}? Doc: {doc.page_content}")
        if "yes" in score.lower():
            filtered.append(doc)
    if not filtered:
        return "web_search"  # Fallback!
    return "generate"

Self-RAG 🪞

Agent reflection tokens use karta hai at each step: Should I retrieve? Is retrieved doc relevant? Is generated answer faithful? Is it useful? Self-corrects at every stage.

graph LR Q[Query] --> D1{Need retrieval?} D1 -->|Yes| R[Retrieve] D1 -->|No| Gen2[Direct Generate] R --> D2{Docs relevant?} D2 -->|Yes| Gen[Generate] D2 -->|No| RW[Rewrite Query] RW --> R Gen --> D3{Faithful?} D3 -->|Yes| A[Final Answer] D3 -->|No| Gen

4. Agentic AI 🦾

RAG se aage — autonomous agents jo reason, plan, execute aur self-correct karte hain.

4.1 What is Agentic AI?

Agent = LLM + Reasoning + Memory + Tools + Action Loop. Simple RAG: single-shot (retrieve once → generate). Agent: multi-turn — reasons what info needed, picks tool, observes result, iterates until done.

Google Interview: "An agent is a control loop where LLM is the controller. Each iteration: perceive state → reason → execute action → update state. Natural language is the intermediate representation for planning."

RAG vs Agentic RAG

4.4 Tool Use & Function Calling 🛠️

LLM decides which tool, generates structured args (JSON). Runtime executes, returns observation.

from langchain_core.tools import tool

@tool
def search_knowledge_base(query: str, top_k: int = 4) -> str:
    """Search internal KB for relevant info."""
    docs = retriever.invoke(query)
    return "\n".join(d.page_content for d in docs[:top_k])

@tool  
def execute_sql(query: str) -> str:
    """Execute read-only SQL. Only SELECT allowed."""
    if not query.strip().upper().startswith("SELECT"):
        return "Error: Only SELECT queries permitted"
    return str(db.execute(query).fetchall())

tools = [search_knowledge_base, execute_sql]
llm_with_tools = llm.bind_tools(tools)
# LLM now decides: search KB? query DB? or answer directly?

5. Agentic Workflows — LangGraph 🕸️

LangGraph = nodes (functions) + edges (decisions) + state (shared memory). Cyclic flows possible.

5.1 LangGraph Basics

graph LR S[State] --> N1[Node 1] N1 --> C{Condition} C -->|Path A| N2[Node 2] C -->|Path B| N3[Node 3] N2 --> E[END] N3 --> E

Key Concepts: StateGraph defines the graph. State = TypedDict shared across nodes. Nodes = functions that transform state. Edges = routing (can be conditional). Compile → invoke.

5.2 Key Workflow Patterns

ReAct Pattern

Reason + Act loop. Think → tool call → observe → think again until done.

graph LR I[Input] --> R[Reason] R --> A[Act - Tool] A --> O[Observe] O --> R O --> F[Final Answer]

Plan and Execute

Plan all steps first, execute sequentially, replan if needed. Better for complex multi-step.

graph LR Q[Query] --> P[Planner] P --> X[Executor] X --> RP{Replan?} RP -->|Yes| P RP -->|Done| A[Answer]

Multi-Agent Supervisor

Supervisor delegates to specialized agents (Research, Code, Analysis), collects results.

graph TD U[User] --> Sup[Supervisor] Sup --> RA[Research Agent] Sup --> CA[Code Agent] Sup --> AA[Analysis Agent] RA --> Sup CA --> Sup AA --> Sup Sup --> Ans[Answer]

CRAG with LangGraph

Query → retrieve → grade docs → generate if relevant, else web search fallback.

graph LR Q[Query] --> RN[Retrieve] RN --> GD{Grade Docs} GD -->|Relevant| GEN[Generate] GD -->|Not Relevant| WS[Web Search] WS --> GEN GEN --> Ans[Answer]

5.3 LangGraph Code — Full ReAct Agent

from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode, tools_condition

# 1. State define karo (shared memory)
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]

# 2. Tools define karo
tools = [search_knowledge_base, execute_sql]
llm_with_tools = llm.bind_tools(tools)

# 3. Agent node — reasoning + tool selection
def agent_node(state: AgentState):
    """Agent reasons about query and decides next action."""
    return {"messages": [llm_with_tools.invoke(state["messages"])]}

# 4. Build the graph
graph = StateGraph(AgentState)
graph.add_node("agent", agent_node)
graph.add_node("tools", ToolNode(tools))

# 5. Routing logic
graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", tools_condition)  # Tool needed? → tools node
graph.add_edge("tools", "agent")  # After tool → back to agent (loop!)

# 6. Compile and run
app = graph.compile()
result = app.invoke({"messages": [("user", "What were our Q3 metrics?")]})
print(result["messages"][-1].content)

RAG Cheatsheet

  • Pipeline: Load → Split → Embed → Store → Retrieve → Generate
  • Chunk: Factoid 128-256 | General 256-512 | Complex 512-1024
  • Overlap: 10-20% of chunk size
  • Distance: Cosine (normalize!), Euclidean, Dot Product
  • MMR: lambda 1=relevance, 0=diversity, best 0.5-0.7
  • Hybrid: BM25 + Dense via EnsembleRetriever
  • HyDE: hypothetical answer → embed → search (bridge gap)
  • CRAG: grade docs → web fallback if irrelevant
  • RAG Fusion: multi-query + Reciprocal Rank Fusion
  • Self-RAG: reflection tokens at every step
  • Graph RAG: entities + relations → multi-hop reasoning
  • Parent Doc Retriever: small child match → big parent context
  • Evaluation: RAGAS — faithfulness, relevance, context precision/recall

Agentic AI Cheatsheet

  • Agent = LLM + Memory + Tools + Planning + Loop
  • ReAct: Reason → Act → Observe → repeat
  • Plan & Execute: plan steps → execute → replan
  • LangGraph: nodes=functions, edges=routing, state=shared memory
  • Conditional edges: route based on state (tools_condition)
  • Memory: short-term, long-term, episodic, semantic
  • Supervisor: boss agent → specialized sub-agents
  • HITL: interrupt for human approval on risky actions
  • Tool calling: @tool decorator + llm.bind_tools()
  • StateGraph → add_node → add_edge → compile → invoke
  • Eval: task completion, tool accuracy, faithfulness
  • Production: error handling, retries, timeouts, cost limits
